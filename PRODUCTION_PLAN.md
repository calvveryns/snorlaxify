# План доработки перед production

## Цель

Довести проект до состояния, в котором:
- основной сценарий дедупликации работает end-to-end без ручных заглушек;
- pipeline устойчив к ошибкам и пригоден для эксплуатации;
- качество ML/LLM можно измерять, улучшать и контролировать через воспроизводимый процесс.

Ниже план разделён на два блока:
- критические доработки продукта и платформы;
- улучшение качества ML/LLM части.

---

## 1. Критические доработки

### 1.1. Привести к единому контракту данные между backend, LLM и frontend

**Проблема**

Сейчас есть расхождения в названиях полей и в структуре результата:
- LLM-ответ и frontend ожидают `suggested_name`;
- логика применения результата в БД использует `suggest_name`;
- часть DTO на клиенте и ответы backend описаны непоследовательно.

**Что сделать**

1. Зафиксировать единый JSON-contract для:
   - candidate pair;
   - LLM recommendation;
   - итогового pipeline result;
   - payload для ручного resolve.
2. Обновить backend модели и сериализацию под один формат.
3. Обновить frontend DTO и привязки.
4. Добавить контрактные тесты на сериализацию и десериализацию.

**Результат**

Исключаются ошибки интеграции между pipeline, UI и логикой merge/resolve.

**Оценка**

Приоритет: P0  
Сложность: низкая-средняя  
Срок: 1-2 дня

---

### 1.2. Довести ручную обработку дублей до рабочего сценария

**Проблема**

Страница ручной коррекции дублей работает на моковых данных. Пользователь не может реально просмотреть рекомендации, отредактировать решение и применить его в базу.

**Что сделать**

1. Подключить страницу дублей к реальному API результата задачи.
2. Реализовать загрузку рекомендаций по `task_id`.
3. Реализовать выбор пар для merge/ignore.
4. Реализовать редактирование итогового имени.
5. Реализовать сохранение результата через `resolve` endpoint.
6. Добавить экран успешного применения и обработку ошибок.

**Результат**

Появляется рабочий user flow: запуск pipeline -> просмотр результатов -> подтверждение исправлений -> применение в БД.

**Оценка**

Приоритет: P0  
Сложность: средняя  
Срок: 3-5 дней

---

### 1.3. Завершить основные действия управления pipeline во frontend

**Проблема**

Во frontend есть кнопки и сценарии, которые не работают: pause, restart step, save и часть элементов управления задачами.

**Что сделать**

1. Подключить `pause` и `resume` к backend API.
2. Убрать или скрыть функции, которых backend ещё не поддерживает.
3. Реализовать корректное отображение статусов `pending`, `running`, `paused`, `completed`, `failed`.
4. Добавить автоматический polling с остановкой по terminal status.
5. Добавить обработку ошибок и пустых состояний.

**Результат**

UI перестаёт быть демонстрационным и начинает соответствовать реальным возможностям backend.

**Оценка**

Приоритет: P0  
Сложность: средняя  
Срок: 2-3 дня

---

### 1.4. Ввести аутентификацию и базовую авторизацию

**Проблема**

Сейчас сервис открыт без auth, login-флоу не реализован, CORS открыт полностью.

**Что сделать**

1. Выбрать механизм auth:
   - JWT + session/token API;
   - либо интеграция с корпоративным SSO.
2. Закрыть mutating endpoints авторизацией.
3. Ввести роли минимум:
   - viewer;
   - operator;
   - admin.
4. Ограничить CORS списком доверенных доменов.
5. Реализовать login/logout flow на фронте.
6. Добавить audit trail:
   - кто запустил задачу;
   - кто применил merge;
   - когда и что было изменено.

**Результат**

Сервис становится пригоден для эксплуатации в реальной среде, где важны права доступа и аудит.

**Оценка**

Приоритет: P0  
Сложность: средняя-высокая  
Срок: 5-8 дней

### 1.5. Перевести pipeline с `threading.Thread` на production-grade execution model

**Проблема**

Фоновые задачи сейчас живут в памяти процесса backend. После рестарта состояние управления теряется. Масштабирование и устойчивость ограничены.

**Что сделать**

1. Вынести выполнение pipeline в отдельный worker.
2. Выбрать механизм очереди:
   - Celery;
   - RQ;
   - Dramatiq;
   - Arq.
3. Перевести запуск задач на enqueue model.
4. Хранить state machine задачи в БД.
5. Реализовать корректные переходы состояний:
   - pending;
   - running;
   - paused;
   - completed;
   - failed;
   - canceled.
6. Сделать resume/retry на уровне очереди, а не только in-memory.

**Результат**

Pipeline становится устойчивым к рестартам, пригодным для фонового исполнения и масштабирования.

**Оценка**

Приоритет: P0  
Сложность: высокая  
Срок: 5-10 дней

---

### 1.6. Усилить работу с БД и производительность pipeline

**Проблема**

Сейчас на каждую запись открывается соединение и выполняется отдельный `commit`. Это создаёт узкое место и повышает риск деградации на больших объёмах.

**Что сделать**

1. Ввести connection pooling.
2. Перевести вставку векторов на batch insert/upsert.
3. Добавить chunking для обработки записей.
4. Пересмотреть индексы и структуру таблиц `vector_data`, `pipeline_tasks`, `pipeline_results`.
5. Добавить миграции схемы через Alembic.
6. Ввести политику очистки старых задач и результатов.

**Результат**

Снижается нагрузка на БД и повышается пропускная способность дедупликации.

**Оценка**

Приоритет: P1  
Сложность: средняя-высокая  
Срок: 4-6 дней

---

### 1.7. Добавить наблюдаемость, ошибки и эксплуатационный контур

**Проблема**

Сейчас есть базовые логи, но нет полноценной observability для production.

**Что сделать**

1. Добавить структурированные логи с `task_id`.
2. Добавить таймауты и retry policy для внешних API.
3. Добавить health/readiness checks:
   - БД;
   - embedding API;
   - LLM API.
4. Добавить метрики:
   - длительность шагов pipeline;
   - число обработанных записей;
   - число candidate pairs;
   - доля ошибок;
   - latency внешних сервисов.
5. Настроить алерты на:
   - рост ошибок;
   - зависшие задачи;
   - недоступность внешних моделей.

**Результат**

Проект становится наблюдаемым и управляемым в production.

**Оценка**

Приоритет: P1  
Сложность: средняя  
Срок: 3-5 дней

**Результат**

**Оценка**

Приоритет: P1  
Сложность: средняя

## 2. Улучшение ML/LLM качества

### 2.1. Формализовать ML/LLM contract и параметры inference

**Проблема**

Сейчас интеграции с embedding и LLM API принимают несколько вариантов ответа и не фиксируют стабильный контракт. Это делает поведение хрупким и трудноотлаживаемым.

**Что сделать**

1. Зафиксировать ожидаемый формат embedding response.
2. Зафиксировать ожидаемый формат LLM response.
3. Валидировать ответы через строгие схемы.
4. Ввести явные таймауты, retries и классификацию ошибок:
   - transport error;
   - model error;
   - invalid schema;
   - empty result.
5. Версионировать contract и model settings.

**Результат**

ML/LLM слой становится предсказуемым и проще поддерживается.

**Оценка**

Приоритет: P0  
Сложность: средняя  
Срок: 2-3 дня

---

### 2.2. Собрать эталонный датасет и ввести офлайн-оценку качества

**Проблема**

Сейчас качество дедупликации нельзя измерить. Нет benchmark-набора размеченных дублей и недублей.

**Что сделать**

1. Собрать репрезентативный labelled dataset:
   - true duplicates;
   - hard negatives;
   - похожие, но не дубликаты;
   - разные языковые и орфографические варианты.
2. Разделить набор на train/dev/test для экспериментов.
3. Зафиксировать метрики:
   - precision@candidate stage;
   - recall@candidate stage;
   - final precision;
   - false merge rate;
   - coverage.
4. Сделать отдельный offline evaluation script/report.

**Результат**

Любые изменения в моделях и порогах можно оценивать на фактах, а не вручную.

**Оценка**

Приоритет: P0  
Сложность: средняя-высокая  
Срок: 5-8 дней

---

### 2.3. Оптимизировать candidate generation до вызова LLM

**Проблема**

Сейчас пары выбираются по одному фиксированному порогу расстояния `<= 0.15`. Это может давать либо много шума, либо пропускать полезные кандидаты.

**Что сделать**

1. Вынести пороги и ограничения в конфиг.
2. Протестировать несколько стратегий candidate selection:
   - фиксированный порог;
   - top-k neighbours;
   - top-k + threshold;
   - блокировка по категориям или типам записей.
3. Добавить дополнительные cheap-features до LLM:
   - нормализация текста;
   - token overlap;
   - символьное расстояние;
   - совпадение бренда/артикула/категории.
4. Использовать rule-based prefilter, чтобы не отправлять очевидный шум в LLM.

**Результат**

Снижается стоимость LLM-этапа и растёт качество финальной рекомендации.

**Оценка**

Приоритет: P0  
Сложность: средняя  
Срок: 4-6 дней

---

### 2.4. Улучшить preprocessing и нормализацию входных названий

**Проблема**

Если в embedding идут сырые строки без нормализации, модель может плохо обрабатывать сокращения, регистр, шумовые символы и вариации записи.

**Что сделать**

1. Добавить preprocessing pipeline:
   - trim/whitespace normalization;
   - lowercasing по правилам домена;
   - унификация сокращений;
   - очистка шума;
   - нормализация единиц измерения.
2. Разделить raw value и normalized value.
3. Сравнить качество embeddings по raw vs normalized тексту.
4. При необходимости формировать composite text:
   - название;
   - категория;
   - бренд;
   - артикул.

**Результат**

Повышается устойчивость candidate generation к неаккуратным данным.

**Оценка**

Приоритет: P1  
Сложность: средняя  
Срок: 3-5 дней

---

### 2.5. Переработать prompt и схему ответа LLM

**Проблема**

LLM сейчас получает простой prompt по списку пар. Для production этого мало: не хватает правил принятия решения, ограничений и устойчивости к галлюцинациям.

**Что сделать**

1. Переписать prompt в более строгую rubric-based форму.
2. Явно задать критерии для `high`, `medium`, `low`.
3. Добавить отрицательные инструкции:
   - не придумывать новые пары;
   - не менять входные поля;
   - не заполнять `suggested_name`, если это не duplicate.
4. Ввести few-shot примеры на сложных кейсах.
5. Сравнить single-pair inference vs batched inference.
6. При необходимости разбить задачу на 2 стадии:
   - duplicate classification;
   - canonical name suggestion.

**Результат**

LLM начинает давать более стабильные и лучше контролируемые ответы.

**Оценка**

Приоритет: P0  
Сложность: средняя  
Срок: 3-5 дней

---

### 2.6. Ввести quality gates и human-in-the-loop policy

**Проблема**

Не все рекомендации одинаково безопасно применять автоматически. Нужна политика доверия к модели.

**Что сделать**

1. Ввести confidence policy:
   - auto-merge запрещён по умолчанию;
   - manual review для `medium`;
   - опциональный auto-merge только для очень надёжных кейсов.
2. Сохранить объяснимые признаки решения:
   - distance;
   - normalized forms;
   - дополнительные lexical features;
   - LLM label.
3. Добавить фидбек-цикл:
   - подтверждённый merge;
   - отклонённый merge;
   - исправленное canonical name.
4. Использовать подтверждённые решения как evaluation set для будущих улучшений.

**Результат**

Снижается риск ложных объединений и появляется контур накопления качественной разметки.

**Оценка**

Приоритет: P0  
Сложность: средняя  
Срок: 3-4 дня

---

### 2.7. Провести сравнение моделей и режимов inference

**Проблема**

Сейчас проект зависит от выбранных локальных моделей, но их качество и стоимость не сравниваются системно.

**Что сделать**

1. Протестировать 2-3 embedding модели на одном benchmark.
2. Протестировать 2-3 LLM/инференс-конфигурации.
3. Сравнить:
   - качество;
   - latency;
   - стоимость инфраструктуры;
   - стабильность формата ответа.
4. Зафиксировать recommended baseline.

**Результат**

Выбор модели становится инженерным решением, а не случайной настройкой.

**Оценка**

Приоритет: P1  
Сложность: средняя  
Срок: 4-6 дней

## 4. Рекомендуемый первый спринт

Если брать только самое важное на ближайшую итерацию, рекомендую такой объём:

1. Исправить все расхождения в DTO/JSON-contract.
2. Подключить страницу дублей к реальному backend.
3. Реализовать рабочий `resolve` flow.
4. Доделать pause/resume/status handling на UI.
5. Добавить таймауты и retries для embedding/LLM API.
6. Собрать небольшой размеченный eval-набор и зафиксировать базовые метрики.
7. Переписать prompt и провести первый quality benchmark.

Это даст одновременно:
- рабочий продуктовый сценарий;
- снижение технического риска;
- основу для системной работы над качеством моделей.
